# Roadmap Agent — MVP: план сборки

Flask (JSON API) + Vue 3 + SQLite. Без Telegram, без Docker, без Redis.
Дизайн — тема **Nocturne** из бандла `Roadmap Agent SaaS Design`.

Цель: `pip install -r requirements.txt && python run.py` → рабочее приложение на `localhost:5000`, которое собирает роадмап, рисует граф модулей, ведёт по дневному плану и пересобирает план при `stuck`.

---

## Принципы, которые нельзя нарушать

1. **Приложение обязано работать без интернета и без API-ключей.** Провайдеры LLM и поиска подменяются моками через `.env`. Ключ есть → реальный режим. Ключа нет → демо-режим, но все экраны и вся логика те же.
2. **Фронт — тупой рендерер.** Состояния модулей, коды (`AR-09`), координаты нод, проценты, «stuck 6 days» приходят готовыми из API. Vue ничего не вычисляет.
3. **Расписание считает алгоритм, а не LLM.** Топосорт + жадная раскладка. Модель придумывает структуру и оценивает ресурсы — не более.
4. **LLM не выдаёт URL.** Ссылки приходят только из поиска и проходят HTTP-проверку. Это и есть grounding.

## Этап 1. Скелет и данные (без агента)

**Готово, когда:** `curl /api/roadmaps/1` возвращает граф с координатами и состояниями, ни одного вызова LLM не было.

## Этап 2. Агент: генерация роадмапа

- [ ] `agent/llm.py` — интерфейс + `MockLLM` + `AnthropicLLM` (+ опционально `OllamaLLM`)
- [ ] `agent/search.py` — `MockSearch` + `TavilySearch`
- [ ] `agent/verify.py` — HEAD → GET fallback, таймаут 5 сек, запись `http_status` и `verified_at`
- [ ] `agent/runner.py` — пять шагов прогона, каждый пишет прогресс в `AgentRun.steps`:
      `plan → validate_graph → sourcing → verify → schedule`
- [ ] Валидация DAG: циклы → ретрай в LLM с текстом ошибки (максимум 2 раза)
- [ ] Дедуп ресурсов: нормализованный URL + похожесть заголовков
- [ ] `POST /api/roadmaps` → `202 {roadmap_id, run_id}`, работа в фоновом потоке
- [ ] `GET /api/runs/<id>` → шаги и их статусы (фронт опрашивает раз в 700 мс)

**Готово, когда:** ввёл «Go for backend» → экран сборки честно показал 5 шагов со счётчиками («12 queries · 41 found · 34 alive · 5 dead») → открылась карта с живыми ссылками.

## Этап 3. Жизнь приложения

- [ ] `POST /api/sessions/<id>/complete | skip` — пересчёт состояний модулей
- [ ] `POST /api/modules/<id>/stuck {reason}` → `202 {run_id}` → Replanner
- [ ] Replanner: дробит модуль на 2–4 подмодуля, перевешивает рёбра DAG, добирает ресурсы (другой формат — если человек застрял на тексте, ищем видео), пересобирает расписание, пишет `Insight` → это панель **Plan adapted** на карте
- [ ] `GET /api/roadmaps/<id>/stats` — метрики (SQL), «где застреваешь», инсайты, heatmap на 56 дней
- [ ] Цифры в инсайтах считает SQL, LLM только формулирует текст

**Готово, когда:** пометил модуль stuck → на карте появилась панель «Plan adapted» с тремя подшагами и новыми ресурсами, а дневной план сдвинулся.

## Что резать, если горит время

Режем в этом порядке: heatmap → инсайты → `AnthropicLLM` (остаётся только mock). **Не режем:** экран сборки и replanning — на них держится вся «агентность» проекта, без них это красивый to-do список.
